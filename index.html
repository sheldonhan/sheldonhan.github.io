如何设计高性能架构？

高性能复杂度模型

  1. 单机
    • 计算高性能：
      • 进程模型：多进程、多线程
      • 网络模型：PPC/TPC 常量连接、海量请求，Reactor 海量连接、海量请求
      • 缓存模型：本地缓存 EHCache/OSCache，独立缓存 Redis/MC
    • 存储高性能
      • 存储模型：B+树 读多写少，可变；LSM 写多读少，不可变
  2. 集群
    • 计算高性能：
      • 任务分解：将服务器拆分为不同角色，不同服务器处理不同业务；比任务分配复杂
        • 任务分解案例1：微信服务拆分
        • 任务分解案例2：数据库读写分离（目的是为了提升读的性能）：任务拆分为读+写；MySQL划分为主从
        • 任务分解案例3：Zuul（微服务网关）
      • 任务分配：将任务分配给多个服务器执行
        • 集群高性能任务分配架构设计的关键点：
          • 运行形态：SDK、服务器
          • 配置获取：配置文件、配置中心
          • 分配算法：随机/轮询/权重，Hash/负载，路由/sharding
        • 任务分配案例1：DNS
        • 任务分配案例2：nginx
        • 任务分配案例3：Memchached
    • 存储高性能：任务分解、任务分配

如何设计可扩展架构？

架构设计复杂度模型

  • 业务复杂度
  • 质量复杂度

架构复杂度应对之道

  • 等级1 业务低 质量低 架构模式：框架，例如LAMP，SSH，ROR
  • 等级2 业务高 质量低 架构模式：SOA、DDD、微服务
  • 等级3 业务低 质量低 架构模式：集群、缓存、Reactor、分片、副本
  • 等级4 业务低 质量低 架构模式：融合所有模式

可扩展的复杂度模型

  • 系统适应变化的能力，包含可理解和可复用两个部分
  • 架构可扩展
  • 应用可扩展
  • 代码课扩展

可扩展架构设计-拆分

  • 拆分复杂度模型

可扩展架构设计-封装

  • 预测变化（时间跨度、变化方式） 决定 封装变化
  • 预测的艺术：第一原则：只预测2年内的变化；第二原则：3次法则，预测没有把握就不要封装，等到需要的时候重构即可；

如何做好架构设计？

架构设计三原则介绍

  • 原则1：合适原则

    • 宣言：合适优于业界领先
    • 资源 + 时间 + 业务

  • 原则2：简单原则

    • 宣言：简单优于复杂
    • 可靠性 + 可扩展 + 故障处理效率

  • 原则3：演化原则

    • 宣言：演化优于一步到位
    • 创造 + 迭代优化 + 重构重写

架构设计三原则案例

架构设计三原则应用

  • 设计出来的架构要满足当时的业务需要，符合团队和技术的能力水平（合适原则）
  • 先按照简单的方式来设计架构，然后不断地在实际应用过程中迭代优化（简单原则）
  • 当业务发生变化时，架构要扩展、重构，甚至重写（演化原则）
  • 合适 > 简单 > 演化

架构设计原则常见的判断维度

  • 业务 + 团队 + 技术

什么是面向复杂度架构设计？

架构设计方法论

  • 面向模式软件架构
  • 风险驱动架构设计
  • DDD领域驱动设计：DDD和整洁架构

面向复杂度的架构设计

为什么做架构设计？

  • 核心原因：软件系统规模增长
  • 核心特点：数据结构和算法不再是主要问题，整个系统的结构成为首要问题。

怎么做架构设计？

  • 思路：通过分析系统需求找到系统复杂的地方，然后设计方案

有哪些复杂度？

  • 模式：复杂度来源：高性能、高可用、可扩展、安全、成本

怎么降低复杂度？

  • 套路：分库分表、缓存、集群、分片、微服务、DDD、异地多活……

架构设计环

  • 需求 -> 判断 -> 复杂度 -> 拆解 -> 备选架构 -> 取舍 -> 架构方案(4R) -> 实现
